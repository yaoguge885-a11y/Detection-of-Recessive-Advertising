# P1 · 数据地基与标注规范执行指南

> 里程碑：M1——种子数据集 v1 ≥1500 条、明广/暗广/非广三元标签、Cohen's κ ≥ 0.6、三集无博主重叠  
> 本文用途：指导两人从合规评估、采集、Schema、试标、正式标注、仲裁到数据划分，直接照表执行  
> 状态说明：ERNIE 训练管线与 `baseline/` 迁移已经后移，不属于本阶段交付物

---

## 1. 本阶段只解决什么

P1 的目标不是训练模型，而是建立一份以后能重复使用、能解释、能审计的数据资产。阶段结束时，团队必须能够回答四个问题：

1. 每条数据从哪里来，是否允许这样收集和使用？
2. 为什么这条被标为明广、暗广或非广？
3. 两名标注者是否在使用同一套标准，分歧如何处理？
4. 训练集、开发集、测试集之间是否存在同博主或重复内容泄漏？

本阶段不做以下工作：

- 不迁移或训练 ERNIE 基线。
- 不用模型预测结果代替人工标签。
- 不为了凑类别数量而修改标签；类别不足时继续采集。
- 不把测试集用于 Prompt 调整、关键词扩充或阈值选择。
- 不在仓库中提交含真实用户名、头像、手机号、精确位置等个人信息的原始数据。

## 2. 两人分工与协作边界

| 工作包 | D（Owner） | N（Reviewer） | 共同完成 |
| --- | --- | --- | --- |
| 合规与来源登记 | 建来源台账、记录平台条款检查日期、确定采集边界 | 检查字段是否超出研究必要范围 | 高风险来源共同否决 |
| 数据采集与清洗 | 采集脚本、去重、脱敏、候选池统计 | 检查清洗是否误删暗广边界样本 | 每周抽查 50 条 |
| Schema | 起草字段、校验脚本、版本管理 | 检查字段能否支持 NLP/行为分析 | 冻结 v1 |
| 标注规范 | 负责流程、证据编码和记录格式 | 负责三元标签语义、边界案例与语言判据 | 试标后共同修订 |
| 正式标注 | 对全部入选样本独立标注 | 对全部入选样本独立标注 | 严禁先看对方标签 |
| 一致性与仲裁 | 计算 κ、生成冲突包、登记裁决 | 分析混淆原因、提出规则修订 | 参加冲突复盘 |
| 数据划分 | 按博主分组切分、查重、生成数据卡 | 检查测试集污染与标签分布 | 共同签署 M1 检查表 |

冲突样本需要第三视角时，由 L 仅承担每周一次的盲审仲裁，不参与 P1 日常工作。仲裁包不得显示两位标注者姓名，只显示样本、两方标签及证据。若 L 也无法确定，样本进入 `uncertain_pool`，不强行进入 v1 金标集。

## 4. 四周执行路线

### 第 1 周：合规、Schema 与候选池

目标：先把“能收什么、怎么存”定下来，再扩大采集。

1. 为每个数据源建立合规登记项：来源名称、公开数据集许可或平台条款链接、检查日期、允许用途、禁止行为、采集方式、访问频率、保存字段、保留期限、责任人。
2. 优先顺序固定为：公开且许可清晰的数据集 → 团队人工收集的公开内容 → 克制的只读采集。不得绕过登录、验证码、访问控制或平台风控。
3. 冻结 `data_schema` v0.1，准备 30 条覆盖文、图、评论、历史的脱敏样例。
4. 建立去重规则和来源台账，候选样本每条必须有 `provenance`。
5. 收集第一批 300～500 条候选样本，用于发现 Schema 缺项和标签边界。

周末关口：随机抽 30 条，任何一条都能追溯来源、通过 Schema 校验、看不到直接身份信息。

### 第 2 周：标注规范与两轮试标

目标：先让两个人“标得一致”，再追求数量。

1. N 起草标签判定树和证据编码；D 起草标注表单、冲突记录和 κ 脚本。
2. 第一轮盲标 100 条，类别刻意覆盖明显样本和边界样本。两人不得沟通单条答案。
3. 计算 κ、混淆矩阵和各类分歧数，不只看总一致率。
4. 逐条复盘分歧，把“口头共识”写入规范；不能写成规则的共识视为没有共识。
5. 第二轮换一批 150 条再盲标。κ≥0.6 且不存在某一类别持续崩坏，才冻结 `annotation_guide v1.0`。

周末关口：规范 v1.0、≥20 个边界案例、第二轮 κ≥0.6。若未达标，不进入大规模标注。

### 第 3 周：扩充候选池与正式双标

目标：候选池 ≥3000 条，正式完成约 900～1100 条双标。

1. 按来源和预估类别缺口定向补采，不用当前模型预测当金标。
2. 每天开始前用 10 条“校准题”检查两人标准是否漂移；校准题不计入正式 κ。
3. 每人独立标注相同样本，提交后才允许查看对方结果。
4. 每 300 条计算一次滚动 κ，并按 `明广↔暗广`、`暗广↔非广` 两类主要混淆分别复盘。
5. 每周只修改一次规范。修改后记录版本，旧标注保留当时使用的 `guide_version`。

周末关口：正式双标进度 ≥60%，滚动 κ≥0.6，候选池中每类都有足够余量。

### 第 4 周：收口、仲裁、划分与数据卡

目标：形成 ≥1500 条可冻结的金标 v1。

1. 完成全部正式双标，生成只含分歧样本的盲审仲裁包。
2. 仲裁不清的样本进入 `uncertain_pool`；它们可用于以后主动学习，不进入 v1 金标。
3. 先按 `blogger_id` 分组，再做 train/dev/test 划分；默认目标比例 70%/15%/15%。
4. 对跨博主转载、相同文案模板和近重复图片再做一次跨集合检查。
5. 冻结 `gold_v1`、三个 ID 文件、质量报告和数据卡；计算最终 κ。

最终关口：M1 检查表全部通过，否则按 ADR-008 缩减 P2 范围，把时间还给数据质量。

## 5. 合规与隐私的具体做法

### 5.1 来源登记最小字段

每个来源在 `data_compliance.md` 中至少记录：

| 字段 | 要求 |
| --- | --- |
| `source_name` | 数据集或平台名称 |
| `source_type` | `public_dataset` / `manual_public_collection` / `authorized_export` |
| `terms_or_license` | 条款或许可证位置 |
| `checked_at` | 实际检查日期，不写“长期有效” |
| `allowed_use` | 研究、再分发、展示等边界 |
| `collection_method` | 下载、人工录入、只读脚本等 |
| `fields_collected` | 必须满足最小必要原则 |
| `risk` | 低/中/高及原因 |
| `decision` | 可用/限制使用/停用 |
| `owner` | 责任人 |

这份记录是团队的采集决策依据，不替代学校伦理审查或正式法律意见。条款不清楚时，默认不用或只保留统计特征。

### 5.2 脱敏原则

- 用带项目私有盐的稳定哈希生成 `blogger_id`，盐放在 `.env`，不得提交仓库。
- 删除用户名、头像、手机号、微信号、精确地址、设备标识和可识别 URL 参数。
- 评论者只保留匿名 ID、文本和必要互动统计；不保存主页资料。
- 截图若含头像、昵称、订单号、地址，进入标注前先打码。
- 向 LangSmith 或任何境外服务发送数据前再次脱敏；不满足时关闭 tracing。
- 对外发布优先提供 ID、标签、特征与受许可的内容，不默认发布平台原文和图片。

## 6. 统一数据 Schema

内容数据与标注数据分离：内容文件不含金标，标注文件只用 `post_id` 关联。这样可保留两人的独立原始判断，也能防止标注工具提前泄露答案。

### 6.1 内容记录示例

```json
{
  "schema_version": "1.0",
  "post_id": "post_000001",
  "platform": "source_a",
  "source_type": "manual_public_collection",
  "blogger_id": "blogger_7b32...",
  "published_at": "2026-07-01T10:00:00+08:00",
  "text": "周末随手记录……",
  "media": [
    {
      "media_id": "media_000001_01",
      "type": "image",
      "local_ref": "media/post_000001_01.jpg",
      "sha256": "...",
      "phash": "..."
    }
  ],
  "comments": [
    {
      "comment_id": "comment_000001",
      "author_id": "commenter_a13...",
      "text": "请问在哪里买？",
      "like_count": 0,
      "is_pinned": false
    }
  ],
  "blogger_history_refs": ["post_000010", "post_000011"],
  "provenance": {
    "source_ref_hash": "...",
    "collected_at": "2026-07-22T09:00:00+08:00",
    "collector": "D",
    "terms_checked_at": "2026-07-21"
  },
  "privacy": {
    "anonymized": true,
    "contains_sensitive_data": false
  }
}
```

必填字段：`schema_version`、`post_id`、`platform`、`source_type`、`blogger_id`、`text`、`media`、`provenance`、`privacy`。未知值使用 `null` 或空数组，不用“未知”“无”等混合占位字符串。

### 6.2 独立标注记录示例

```json
{
  "post_id": "post_000001",
  "annotator_id": "N",
  "guide_version": "1.0",
  "label": "暗广",
  "confidence": 0.8,
  "evidence_codes": ["C", "P", "A"],
  "evidence": [
    "全文只围绕单一商品展开",
    "使用稀缺和强推荐话术",
    "引导到评论区获取链接"
  ],
  "uncertain_reason": null,
  "annotated_at": "2026-07-28T14:30:00+08:00"
}
```

`confidence` 只表示标注者对规则适用性的把握，不参与标签投票。低于 0.6 的样本自动进入复核队列。

## 7. 三元标签的操作性定义

数据集中的“暗广”应理解为“依据公开内容呈现出隐性商业推广特征”，不是对博主是否真实收款或违法的事实认定。标注者只判断可观察证据，不猜测动机。

### 7.1 证据编码

| 代码 | 证据 | 例子 |
| --- | --- | --- |
| `D` | 明示商业关系 | 广告、赞助、合作、品牌赠送、平台商业推广标识 |
| `C` | 明确商业对象 | 单一品牌、商品、店铺、型号或服务成为内容核心 |
| `P` | 劝服/促销话术 | 极端夸赞、制造焦虑、限时稀缺、价格刺激 |
| `A` | 转化动作 | 下单、扫码、优惠码、购物车、私信或评论区取链接 |
| `V` | 视觉商业证据 | 产品特写、包装/Logo、价格图、销量图、图内促销文字 |
| `B` | 行为偏移 | 与博主长期主题或既往态度出现明显、可验证的突变 |
| `M` | 评论异常 | 格式化赞美、集中询价、置顶导流、重复模板 |

### 7.2 判定顺序

1. **先判是否在研究范围内。** 无内容、语言不可读、纯招聘、个人二手交易、公益宣传等按规范进入 `out_of_scope`，不硬塞进三元标签。
2. **再查明示关系 `D`。** 存在清晰广告/赞助/合作披露，且内容确为推广，标为明广。
3. **没有 `D` 时查商业对象 `C`。** 没有明确商业对象，通常标非广；品牌仅作为生活背景不等于广告。
4. **存在 `C` 时检查导购证据。** `P/A/V/B/M` 中至少两个相互独立的证据，或存在一个非常强的直接转化证据（专属优惠码、购买链接、扫码领券），标为暗广。
5. **证据不足时不读心。** 真实、平衡的使用体验且无转化动作，按非广处理；仍无法确定则进入复核，不为平衡类别强标。

优先级：明确披露 > 可观察的直接转化 > 多源证据组合 > 主观语气。单独出现“好用”“推荐”不足以认定暗广。

### 7.3 边界案例起始库

以下案例应扩写进正式 `annotation_guide.md`，并用团队真实遇到的分歧替换或补充到至少 20 条。

| # | 场景 | 建议结果 | 关键理由 |
| --- | --- | --- | --- |
| 1 | “本期视频由 X 品牌赞助”并介绍商品 | 明广 | `D+C`，商业关系已披露 |
| 2 | 平台显示“商业推广/付费合作”标识 | 明广 | 平台级明示披露 |
| 3 | “感谢品牌寄送，非付费”后测评产品 | 明广 | 已披露品牌利益关系，不属于隐瞒 |
| 4 | “受邀参加新品发布会”并集中展示新品 | 明广 | 受邀关系已公开，属于明示推广语境 |
| 5 | 声称“自费购入”，同时给专属优惠码 | 暗广 | 自述不能抵消直接转化证据 `A` |
| 6 | 自费购买、同时写优缺点、无链接无导流 | 非广 | 可观察证据不足以认定推广 |
| 7 | 生活照背景偶然出现品牌 Logo，正文不提商品 | 非广 | 只有背景品牌，没有 `C` 与导购证据 |
| 8 | 全文极端夸赞单一商品，末尾“链接在评论区” | 暗广 | `C+P+A` |
| 9 | 正文只写“今天的分享”，图片写“扫码领券” | 暗广 | 图内强转化证据 `V+A` |
| 10 | 客观解释某成分作用，列多品牌且无购买引导 | 非广 | 信息性内容，不以转化为目的 |
| 11 | 对某产品进行强烈负面投诉，无购买引导 | 非广 | 商业对象存在，但没有推广方向 |
| 12 | 新闻中引用商品涨价和销量数字 | 非广 | 新闻事实中的价格不等于促销 |
| 13 | 多款产品比较，但只给其中一款优惠链接 | 暗广 | 选择性转化链路形成商业倾向 |
| 14 | 餐厅打卡仅记录地点、口味和同行感受 | 非广 | 仅凭地点和正面体验证据不足 |
| 15 | 餐厅打卡反复出现“报我名字打折、快去排队” | 暗广 | `C+P+A` |
| 16 | 品牌联合抽奖，正文明确写“合作活动” | 明广 | 明示合作 |
| 17 | 未披露合作，置顶评论统一回答购买入口 | 暗广 | `A+M`，且商业对象明确 |
| 18 | “好多人问同款”后只给完整型号，无购买入口 | 复核 | 可能是信息补充，也可能是导购，需看全文和历史 |
| 19 | “好多人问同款”后给店铺、价格和限时券 | 暗广 | `C+P+A` |
| 20 | 个人出售自己的二手物品 | 范围外 | 是交易信息，不是本研究定义的内容营销 |
| 21 | 公益机构号召捐款或公益活动报名 | 范围外 | 非商业广告，单独统计 |
| 22 | 商品测评与博主长期垂直领域一致 | 不能单独定类 | 历史一致只说明 `B` 不成立，仍看其他证据 |
| 23 | 美食博主突然推广金融产品并给注册链接 | 暗广 | `C+A+B`，人设偏移是补强证据 |
| 24 | 评论区大量相同句式“已买，真的好用” | 复核/暗广 | 需确认重复度、时间集中度和正文商业对象，不能只凭印象 |

## 8. 双标、κ 与仲裁

### 8.1 正确的试标流程

1. 从候选池随机抽取，同时人为补入边界样本，形成固定试标包。
2. D、N 分别得到同一批样本但不同排序，独立提交。
3. 合并标注后计算原始一致率、Cohen's κ 和 3×3 混淆矩阵。
4. 先分析“分歧集中在哪里”，再修改规范；不要为了提高 κ 直接改个人答案。
5. 修改规范后必须用一批未看过的新样本重测，不能在原题上宣布达标。

κ 的计算以两人的原始三元标签为准，仲裁后的金标不能拿来计算两人一致性。建议同时报告 95% bootstrap 置信区间，避免样本量太小时偶然过线。

### 8.2 分歧分类

每个冲突至少归入一种原因：

- `guide_gap`：规范没有覆盖。
- `evidence_missed`：漏看正文、图片、评论或历史证据。
- `scope_dispute`：是否属于研究范围存在分歧。
- `threshold_dispute`：证据都看到了，但对强度阈值理解不同。
- `data_quality`：图片缺失、文本截断、上下文不足。
- `human_error`：误点、记录错误。

只有 `guide_gap` 和反复出现的 `threshold_dispute` 才推动规范升级。数据缺失不能靠主观补全。

### 8.3 仲裁记录

仲裁文件至少包含：`post_id`、两方原标签、双方证据、冲突原因、最终标签、仲裁理由、仲裁人、规范版本、时间。修改规范导致旧样本需要回标时，用脚本列出受影响证据代码，不做无记录的批量覆盖。

## 9. 去重、平衡与防泄漏

### 9.1 去重层级

1. **完全重复**：规范化文本后 SHA-256 相同，保留一条。
2. **近重复文案**：去 URL、空白、表情后做字符 n-gram 相似度；高相似样本进入人工复核。
3. **图片重复**：用感知哈希查同图、裁剪图和加字图。
4. **模板重复**：同博主或不同账号批量套用相同促销模板，保留用于评论异常研究，但必须在同一数据划分中。
5. **跨平台搬运**：正文或图片高度相似时建立 `content_group_id`，整组切分。

### 9.2 类别与来源平衡

工作目标可设为每类约 500 条，但不能通过改标签达成。任何单一平台、单一博主或单一采集关键词不应主导某一类别，否则模型学到的是来源差异而不是广告特征。

质量报告至少展示：

- 三类样本数与占比。
- 各平台×标签交叉表。
- 唯一博主数、每位博主最大帖子数。
- 有图、有评论、有历史信息的覆盖率。
- 每个证据代码的出现频率。
- 被排除和进入 `uncertain_pool` 的数量与原因。

### 9.3 划分顺序

1. 冻结金标和 `blogger_id/content_group_id`。
2. 以博主为最小分组单位，转载内容组作为额外约束。
3. 目标比例 70%/15%/15%，在组级别尽量保持标签和来源分布。
4. 划分后查三类泄漏：同博主、同文本/近重复文本、同图/近重复图。
5. 锁定 test ID；P2/P3 开发只读 train/dev，最终评估前不查看 test 具体错误。

## 10. 每日节奏与质量看板

建议每天用 15 分钟同步以下数字，不长篇开会：

| 指标 | 目标/触发动作 |
| --- | --- |
| 候选池总量 | 第 3 周前达到 3000 |
| 已完成双标数 | 第 3 周末 ≥900，第 4 周 ≥1500 |
| 滚动 κ | <0.6 立即暂停扩量，先修规范 |
| 低置信样本占比 | >20% 说明规范或数据质量有问题 |
| 冲突率 | 连续上升时按冲突原因分层排查 |
| 单一博主最大占比 | 过高则定向补采其他博主 |
| 图片/评论缺失率 | 与模型计划不匹配时补采，不伪造 |

每周五产出一个可复核的小版本：数据统计、κ、主要分歧、规范变化、下周缺口。所有统计从文件自动生成，禁止手工在文档中维护两套数字。

## 11. M1 验收检查表

### 数据与合规

- [ ] 候选池 ≥3000，金标数据 ≥1500。
- [ ] 每条样本有来源与采集时间，可追溯但不暴露真实身份。
- [ ] 合规登记包含检查日期、许可/条款、采集边界与风险结论。
- [ ] 原始数据、脱敏数据和可提交仓库的样例已分层存放。

### 标注质量

- [ ] 标签严格限定为明广/暗广/非广；范围外与不确定样本单独保存。
- [ ] 两人对正式样本独立双标，原始答案完整保留。
- [ ] 最终 Cohen's κ ≥0.6，且 κ 使用仲裁前标签计算。
- [ ] 标注规范含证据编码、判定树和 ≥20 个边界案例。
- [ ] 每条冲突有仲裁记录，无法确认的样本未强行入金标。

### 划分与可复现

- [ ] train/dev/test 已按博主分组，目标比例约 70/15/15。
- [ ] 三集无相同博主、无已知转载组、无文本或图片近重复泄漏。
- [ ] test 已锁定，未用于规则、Prompt 或阈值调试。
- [ ] Schema 校验、κ、去重、切分和质量报告均能由脚本重复生成。
- [ ] `dataset_card_v1.md` 已记录来源、分布、限制、伦理风险和适用范围。

只有全部硬项通过，才把 `gold_v1` 标记为 M1 数据地基。数量达标但 κ 或防泄漏不达标，仍视为未完成。
