# P2 · 工具舱：模型工具化执行指南

> 里程碑：M2——至少 6 个工具可独立调用、统一结构化接口、返回证据、单测全绿、预期异常不崩  
> 本文用途：指导两人把现有模型能力封装成稳定工具，并补齐计划表中的 P2 工具  
> 最新状态：YOLO11 + EasyOCR 的真实视觉分析链路已经跑通；ERNIE `baseline/` 迁移后移，不属于 P2

---

## 1. P2 的正确目标

P2 不是继续堆模型，而是把“能跑一次的能力”变成“任何智能体都能稳定调用的工具”。每个工具都必须做到：

1. 输入字段清楚，调用者不用了解底层模型。
2. 输出是可校验、可序列化的结构化数据，而不是一段随意文本。
3. 除了分数或结论，还返回可进入审查报告的证据。
4. 无图、空历史、缺依赖、模型超时等预期情况不会拖垮整张 LangGraph。
5. 核心逻辑可以脱离 LangGraph 独立测试，换模型时不改上层接口。

已有资产应直接复用：

- `impad/tools/vision.py`：YOLO11 物体检测、EasyOCR、焦点计算，且模型有模块级缓存。
- `impad/agents/vision_agent.py`：视觉结果转投票、四级降级逻辑。
- `impad/tools/keywords.py`：6 维确定性关键词特征、证据命中。
- `impad/agents/nlp_agent.py`：文本 LLM 结构化输出及无 Key 规则降级。

禁止为了“工具化”复制一套 YOLO/OCR 或关键词实现。P2 应增加薄封装、公共契约和新能力，让现有核心函数成为唯一事实来源。

本阶段不做：

- 不迁移、训练或接入 ERNIE 基线。
- 不重写已经跑通的多智能体图。
- 不上 FastMCP 或 A2A；按 ADR-007，P2 使用 LangChain 原生 `@tool` + Pydantic。
- 不承诺当前 COCO YOLO 能识别品牌 Logo；普通物体检测和品牌识别必须在输出中区分。
- 不把工具的模型预测写回 P1 人工金标，避免标注锚定和标签污染。

## 2. 两人分工

| 工作包 | L（Owner） | V（Reviewer） |
| --- | --- | --- |
| 公共契约 | 定 Pydantic 输入/输出、状态码、证据结构、注册表 | 用视觉工具验证契约能表达 bbox/OCR/模型降级 |
| 文本能力 | `analyze_text_intent`、`sentiment_curve` | 检查证据能否与视觉结果对齐 |
| 历史/评论能力 | `topic_drift`、`comment_anomaly` | 复核空历史、少评论等降级场景 |
| 视觉能力 | 评审接口、不重复实现视觉核心 | `ocr_extract`、`detect_logo_product`、`image_text_consistency` |
| 测试基础设施 | 单元测试规范、mock LLM、工具注册表测试 | 固定图片集、视觉真实链路 smoke test |
| 集成与文档 | 工具清单、统一调用示例、全套回归 | 视觉质量报告、性能记录 |

共同规则：每个工具由一人实现，另一人至少审一遍“输入输出、证据、降级、测试”四项。不能由作者本人单独宣布完成。

## 3. M2 工具清单与优先级

计划表有 7 个业务工具，M2 硬门槛是至少 6 个。当前建议全部实现 v1，其中前 6 个为主线，`comment_anomaly` 可在时间不足时降级为简单规则版。

| # | 工具名 | Owner | 当前基础 | P2 v1 要补齐的内容 | M2 优先级 |
| --- | --- | --- | --- | --- | --- |
| 2.1 | `analyze_text_intent` | L | NLP Agent + 6 维关键词 | 独立工具、意图分、话术模式、文本证据 span | 必做 |
| 2.2 | `sentiment_curve` | L | 无独立工具 | 当前情感、焦虑/紧迫指数、序列突变 | 必做 |
| 2.3 | `ocr_extract` | V | EasyOCR 已真实跑通 | 独立工具、全文、文本块、二维码/销量字段、降级 | 必做 |
| 2.4 | `image_text_consistency` | V | YOLO/OCR 结构化结果 | 一致性分、关系类型、跨模态证据 | 必做 |
| 2.5 | `detect_logo_product` | V | YOLO COCO 物体检测已跑通 | 商品清单；Logo/品牌能力边界；OCR/VLM 补强 | 必做 |
| 2.6 | `topic_drift` | L | 行为 Agent 仍为占位 | 当前帖与历史主题距离、相似历史、漂移证据 | 必做 |
| 2.7 | `comment_anomaly` | L | 行为 Agent 简单规则 | 重复、突发、格式化赞美等异常度 | 应做 |

如果 M1 未达标并触发 ADR-008，P2 压缩时优先保留 2.1、2.3、2.4、2.6 和公共接口，再从 2.2/2.5/2.7 中选择能最快达到“总计 6 个”的工具。由于当前 YOLO 已跑通，2.5 的成本已经较低，不应无理由重做或删除。

## 5. 统一工具契约

### 5.1 两层实现，避免 `@tool` 绑死业务逻辑

每个工具分为两层：

1. `_xxx_core()`：普通 Python 函数，接收 Pydantic 输入，返回 Pydantic Result。Agent、API 和单测都可以直接调用。
2. `@tool xxx()`：很薄的 LangChain 适配层，只负责参数校验、调用 core、返回 JSON 可序列化字典。

这样以后换模型、接 FastMCP 或让已有 Agent 调用时，不需要从 `StructuredTool` 对象中反向拆业务逻辑。

示意骨架：

```python
from typing import Literal
from pydantic import BaseModel, Field
from langchain_core.tools import tool


class TextIntentInput(BaseModel):
    text: str = Field(min_length=1)
    comments: list[str] = Field(default_factory=list)


class ToolEvidence(BaseModel):
    kind: str
    source: str
    quote: str | None = None
    score: float | None = None
    bbox: list[int] | None = None


class TextIntentResult(BaseModel):
    tool_name: Literal["analyze_text_intent"] = "analyze_text_intent"
    tool_version: str = "1.0"
    status: Literal["ok", "degraded", "skipped", "error"]
    score: float | None = None
    evidence: list[ToolEvidence] = Field(default_factory=list)
    warnings: list[str] = Field(default_factory=list)
    payload: dict = Field(default_factory=dict)


def _analyze_text_intent_core(inp: TextIntentInput) -> TextIntentResult:
    ...


@tool(args_schema=TextIntentInput)
def analyze_text_intent(text: str, comments: list[str] | None = None) -> dict:
    """分析帖子商业意图与劝服话术，返回分数和文本证据。"""
    result = _analyze_text_intent_core(
        TextIntentInput(text=text, comments=comments or [])
    )
    return result.model_dump()
```

上面的短例只展示接口形状；正式实现还应补充字段说明与取值约束。

### 5.2 公共状态语义

| `status` | 什么时候用 | 是否可参与后续判断 |
| --- | --- | --- |
| `ok` | 主实现正常完成 | 可以 |
| `degraded` | 主模型不可用，已用规则/较弱模型产出 | 可以，但下调可靠度并写 warning |
| `skipped` | 缺少必要输入，如无图、历史少于阈值 | 不参与，分数应为 `null` |
| `error` | 输入存在但执行失败，且没有安全替代 | 不参与；返回错误类型，不泄露 Key/堆栈 |

公共输出至少包含：

- `tool_name`：固定英文名。
- `tool_version`：接口或算法改变时升级。
- `status`：四态之一。
- `score`：含义由工具文档明确，统一归一化到 0～1；不适用时为 `null`。
- `evidence`：结构化证据列表，不得只给一段总结。
- `warnings`：降级、输入不足、模型边界。
- `payload`：该工具专属结果。
- `model_info`：可选，记录 provider、model、规则版本；不得包含 API Key。

证据使用统一结构：文本证据给 `quote/span`，视觉证据给 `bbox`，历史证据给 `related_post_id`，评论证据给 `comment_ids`。Judge 以后只读证据对象，不解析模型自然语言。

### 5.3 分数方向必须固定

- `analyze_text_intent.score`：越高越有商业导购意图。
- `sentiment_curve.score`：定义为焦虑/劝服情绪强度，而不是普通正负情感；普通情感另放 payload。
- `image_text_consistency.score`：越高表示语义越一致，不直接代表越像广告。
- `detect_logo_product.score`：商品/品牌商业显著度，不是三元广告概率。
- `topic_drift.score`：越高表示越偏离历史主题。
- `comment_anomaly.score`：越高表示评论行为越异常。

禁止把所有分数都解释成“暗广概率”。这些是证据特征，最终三分类由 Agent/Judge 聚合。

## 6. 七个工具的具体做法

### 6.1 `analyze_text_intent`

**输入**：`text`、可选 `comments`、可选语言。  
**输出 payload**：`commercial_intent_score`、`persuasion_patterns`、`keyword_weights`、`matched_spans`、`summary`。

实现顺序：

1. 先调用现有 `compute_keyword_weights()` 与 `keyword_hits()`，保证任何环境都有确定性证据。
2. 配置 LLM 时，复用 `nlp_agent.py` 的三分类判据，但工具只输出商业意图、话术模式和证据，不直接冒充最终 Judge。
3. LLM 继续采用 ADR-010：`json_mode`、英文字段名、Pydantic 校验。
4. 无 Key、超时或 JSON 校验失败时，使用关键词权重和 `ad_pressure()` 降级，`status="degraded"`。
5. `matched_spans` 保留原文命中片段和起止位置，避免报告只能说“模型认为可疑”。

最低测试：明显明广、典型暗广、生活记录、空文本校验、无 Key 降级、LLM 非法 JSON、证据 span 可回指原文。

完成标准：分数 0～1、至少一种证据、相同输入的规则降级结果确定、输出不含最终整图结论。

### 6.2 `sentiment_curve`

**输入**：当前文本；可选 `history=[{post_id, text, published_at}]`。  
**输出 payload**：`current_sentiment`（正/中/负及概率）、`anxiety_score`、`urgency_score`、`curve_points`、`change_points`。

工作方法：

1. v1 可用中文情感模型或 LLM，不需要训练新模型；调用层保持厂商无关。
2. 焦虑与紧迫不是普通负面情感。单独检测“怕错过、容貌/健康焦虑、库存稀缺、限时”等劝服模式。
3. 有至少 3 条按时间排序的历史文本才生成曲线；不足时仍返回当前点，但 `change_points=[]` 并给 warning。
4. 突变可先用相邻点差值或滑动 z-score；阈值放配置，不写死在 Prompt。
5. 无模型时用小型情感/紧迫词表降级，并明确 `model_info="rule_v1"`。

最低测试：全正面、明显焦虑促销、普通负面吐槽、历史顺序打乱、单点历史、时间缺失、模型超时。

完成标准：输出曲线长度与有效输入一致；当前情感和焦虑/紧迫分开；无历史不报错。

### 6.3 `ocr_extract`

**输入**：本地 `image_path`；可选语言和 OCR 置信度阈值。  
**输出 payload**：`full_text`、`text_blocks[{text, confidence, bbox}]`、`qr_codes`、`sales_signals`。

工作方法：

1. 直接复用 `vision.py` 的 `ImageAnalyzer`/`analyze_image()`，不要再初始化一套 EasyOCR。
2. 把现有 `texts` 原样转换成工具契约，保留 bbox 和置信度；`full_text` 按阅读顺序拼接。
3. 二维码可用 OpenCV `QRCodeDetector` 做轻量检测/解码。解不出内容也可返回 `detected=true`，但不得伪造 URL。
4. 对 OCR 文本用正则提取价格、销量、折扣、优惠码等 `sales_signals`，每个信号必须保留原始文本块。
5. 无图为 `skipped`；文件不存在为 `error` 或项目统一的 `skipped`，两者选定后全项目一致；缺 EasyOCR 时若 YOLO 可用但无 OCR，返回 `degraded`。

注意：工具层默认只接收本地受控路径。远程 URL 的下载、类型/大小校验和 SSRF 防护应放在独立输入层，不能让 OCR 工具直接请求任意 URL。

最低测试：中文促销图、英文短图、无文字图、二维码图、低置信块过滤、文件不存在、OCR 依赖被 mock 为缺失。

完成标准：固定中文测试集可重复提取；每个文本块有 bbox；缺依赖和无文字均不抛出未处理异常。

### 6.4 `image_text_consistency`

**输入**：帖子正文、`image_path`；可选已计算的 `vision_context`。  
**输出 payload**：`consistency_score`、`relation`、`reason`、`aligned_evidence`、`conflicting_evidence`。

`relation` 固定为：`aligned`（一致）、`complementary`（互补）、`conflicting`（冲突）、`insufficient`（信息不足）。

实现路线：

1. 优先复用已经计算的 YOLO/OCR 结果；同一图片不得为三个视觉工具重复跑三次模型。
2. v1 主实现可选 Qwen-VL 或 CLIP。若用 VLM，强制结构化输出并限制只基于可见内容作答。
3. 无 VLM 时，可用“正文与 OCR 文本的 embedding 相似度 + 正文实体与 YOLO 物体映射”作为降级，但必须标记 `degraded`；它不是完整图像语义理解。
4. 证据至少说明：正文声称的核心对象、图片/OCR 显示的核心对象、两者的对应或冲突。
5. 一致性高不等于广告；例如普通旅行记录图文一致。该工具不得直接输出明广/暗广/非广。

质量小测：准备至少 30 对人工构造或许可清晰的图文，其中包含匹配、互补、冲突和信息不足。要求匹配组平均分明显高于随机错配组，并记录均值差，不只展示几个成功案例。

最低测试：一致图文、随机错配、正文空、图片无对象无文字、VLM 超时、复用已有 `vision_context` 时不再次调用 YOLO。

### 6.5 `detect_logo_product`

**输入**：`image_path`；可选 `vision_context`。  
**输出 payload**：`objects`、`commercial_objects`、`brand_candidates`、`logo_candidates`、`capabilities`。

重要能力边界：当前 YOLO11 COCO 预训练模型能识别 bottle、cell phone、laptop 等通用物体，不能据此声称识别出具体品牌 Logo。v1 应诚实区分：

- `objects`：YOLO 真实检测到的通用类别、置信度、bbox。
- `commercial_objects`：现有 `commercial_objects()` 过滤出的带货弱信号。
- `brand_candidates`：从 OCR 包装文字或可选 VLM 得到的品牌候选，并标来源。
- `logo_candidates`：只有专门 Logo 模型或 VLM 明确产出时才填写。
- `capabilities.logo_detection`：`none` / `vlm_candidate` / `dedicated_model`。

先把现有 YOLO 能力稳定封装即可，不为满足工具名称虚构 Logo 精度。后续若换专用模型，只替换 core，接口保持不变。

最低测试：含商品图、非商品生活图、多物体图、空检测、文件缺失、YOLO 异常、OCR 品牌候选与 bbox 来源。

### 6.6 `topic_drift`

**输入**：当前帖 `post_id/text`；至少 3 条同博主历史帖。  
**输出 payload**：`drift_score`、`current_topic`、`history_profile`、`nearest_posts`、`distance_details`。

工作方法：

1. P2 v1 优先使用中文 embedding（ADR 中的 `bge-small-zh-v1.5`）计算当前向量与历史中心/近邻的余弦距离，不必先上 BERTopic 重依赖。
2. 历史只使用当前帖发布时间之前的数据，避免时间穿越。
3. 历史按最近 N 条或时间窗选取，并在结果记录窗口大小。
4. 漂移分可由“到历史中心距离 + 最近邻距离”组合得到，公式和归一化参数写入模块文档。
5. 少于 3 条历史、正文为空或 embedding 不可用时返回 `skipped/degraded`，不得用 0 冒充“没有漂移”。
6. BERTopic + JS/KL 散度可作为 v2 替换项，接口不变。

最低测试：同主题历史、完全无关主题、历史中混有少量异常、历史不足、未来帖子过滤、embedding 服务失败。

完成标准：在合成 sanity set 中，同主题漂移分低于明显跨领域样本；返回最相近历史证据，而非只有一个分数。

### 6.7 `comment_anomaly`

**输入**：评论列表，建议包含 `comment_id/text/created_at/author_id/like_count/is_pinned`。  
**输出 payload**：`anomaly_score`、`duplicate_ratio`、`burst_score`、`praise_template_score`、`suspicious_comment_ids`、`signals`。

v1 先做可解释规则与聚类，不上 GNN：

1. 规范化评论后计算完全重复和近重复比例。
2. 有时间戳时检测短时间突发；没有时间戳时跳过该特征，不填 0 误导。
3. 检测高度相似的格式化赞美、集中询价和重复购买见证。
4. 结合置顶评论、同作者重复、文本长度过度一致等弱信号。
5. 各分项归一化后加权，权重放配置，并在证据中列出触发的评论 ID。

最低测试：正常讨论、复制粘贴评论、短时突发、评论太少、缺时间戳、空评论、全 emoji。

完成标准：至少 5 条有效评论才计算总异常度；输入不足返回 `skipped`；任何异常分都能追溯到具体评论。

## 7. 视觉推理复用与缓存

YOLO/OCR 已跑通后，P2 最大的工程风险是重复推理。建议建立一次分析、多工具复用的 `VisionContext`：

```json
{
  "image_id": "sha256:...",
  "model_version": "yolo11n+easyocr",
  "objects": [],
  "texts": [],
  "focus": {},
  "created_at": "..."
}
```

规则：

- 缓存键至少包含图片内容哈希、模型版本和关键阈值，不能只用文件路径。
- `ocr_extract`、`detect_logo_product`、`image_text_consistency` 优先接收 `vision_context`。
- 缓存存模型原始结构化发现，不缓存最终广告结论。
- 测试必须断言复用 context 时不会再次调用 `analyze_image()`。
- 结果缓存与模块级模型缓存是两回事：现有代码只保证模型不重复加载，不保证同一图片不重复推理。

## 8. 降级与错误处理矩阵

| 场景 | 工具行为 | 状态 | 后续影响 |
| --- | --- | --- | --- |
| 空正文 | 文本工具不分析，返回原因 | `skipped` | Judge 忽略 |
| 无图片 | 三个视觉工具返回无图原因 | `skipped` | Judge 忽略 |
| 图片路径不存在 | 返回安全错误，不带本机绝对敏感路径 | `error` | 不拖垮图 |
| 视觉依赖缺失 | 返回缺依赖说明；有可用弱能力则降级 | `degraded/skipped` | 降低可靠度 |
| 无博主历史 | `topic_drift` 不计算 | `skipped` | 不能用 0 表示无漂移 |
| 评论不足 | `comment_anomaly` 不计算 | `skipped` | 不能用 0 表示正常 |
| LLM/VLM 超时 | 最多一次短重试，随后规则/本地模型降级 | `degraded` | 记录 warning |
| LLM JSON 不合法 | Pydantic 校验失败后降级 | `degraded` | 保存错误类型，不保存原始敏感响应 |
| 模型推理异常 | 工具边界捕获并返回错误类型 | `error` | Agent 继续运行 |

工具不得使用裸 `except: pass`。错误日志可以保留堆栈，返回给 Agent 的结果只保留安全的错误类别和简短原因。

## 9. 测试方法

### 9.1 每个工具的最小测试模板

每个工具至少有以下测试：

1. 正常输入返回 `status="ok"`，且通过 Result 模型校验。
2. 空值/缺字段得到约定的 `skipped` 或输入校验错误。
3. 底层依赖缺失时按矩阵降级。
4. 底层模型抛异常时工具自身不崩。
5. `evidence` 非空且能回指输入。
6. 所有字段可 `json.dumps()`，不返回 numpy、Path、模型对象等不可序列化类型。
7. 分数为 `null` 或落在 0～1。
8. 相同的规则输入得到确定结果。

### 9.2 单元测试与真实模型测试分开

默认回归测试必须：零 API Key、零联网、可在未安装视觉重依赖的电脑运行。通过 monkeypatch/mock 注入固定模型输出。

真实模型测试单独标记为 integration：

```powershell
cd implicit-ad-agent
python -m pytest -q
python -m pytest -m vision_integration -q
```

需要在 `pyproject.toml` 注册 marker，避免 pytest warning。真实视觉测试使用仓库内许可清晰、已脱敏的小图片；LLM/VLM integration 先跑 10 条固定样本，确认 Prompt 和成本后再扩量。

### 9.3 工具注册表验收

`registry.py` 对外暴露固定列表，例如：

```python
TOOLS_V1 = [
    analyze_text_intent,
    sentiment_curve,
    ocr_extract,
    image_text_consistency,
    detect_logo_product,
    topic_drift,
    comment_anomaly,
]
```

注册表测试应自动检查：

- 工具名唯一且与计划表一致。
- 每个工具有非空 docstring，明确何时调用。
- 每个工具有 Pydantic args schema。
- 每个工具可以用最小样例独立 invoke。
- 至少 6 个工具达到 M2 的 `ready=true`，不是只有空壳函数。

## 12. 代码协作与提交纪律

1. 两人都从最新主分支切自己的功能分支，禁止在新文件夹重新 `git init`。
2. 公共 `contracts.py` 先由两人共同冻结，再并行写工具，避免每天解决 merge 冲突。
3. 一个提交只完成一个可描述的动作，例如“封装 OCR 工具并加无图降级测试”。
4. 外部模型调用必须 mock；API Key 只放 `.env`。
5. Windows 下统一使用 `python -m pip`、`python -m pytest`，不依赖可能损坏的裸 launcher。
6. 修改已有 `vision.py`、`keywords.py` 前先确认是否能通过薄封装完成；核心改动必须跑原有 Agent 测试，防止破坏现有链路。
